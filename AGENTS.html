This file provides scoped instructions for AI coding assistants working under blog/.

技术博客写作约束 ​

blog/ 目录下的所有内容都应该保持高信息密度：

  • 尽量减少低信息密度的过渡句、重复解释和空泛表述
  • 优先把关键信息直接写进标题、列表、表格、代码注释和配置注释中
  • 能用一行注释说清的内容，不要再在代码块后重复解释一遍
  • 示例要尽量“可直接使用”，避免为了凑篇幅添加冗余文字
  • 在不影响可读性的前提下，优先压缩篇幅、提高单位段落的信息量

Build, Lint, and Test Commands ​

blog/ 目录对应的文档站点使用 package.json 中的 docs:* 命令进行验证。目前可用命令：

  • pnpm docs:dev：启动 VitePress 开发服务器，适合边改边看
  • pnpm docs:build：执行生产构建，检查文档站点是否可以成功构建
  • pnpm docs:preview：预览构建产物，适合检查最终页面效果

每次修改 blog/ 下的内容后，都应该至少执行一次相关验证命令：

  • 优先使用 package.json 中已有的 docs:* 命令进行验证
  • 能运行构建时，至少执行一次 pnpm docs:build
  • 如果本次修改更适合本地预览，可以补充执行 pnpm docs:dev
  • 如果需要检查构建后的最终页面表现，可以补充执行 pnpm docs:preview
  • 如果因为环境、依赖或时间限制无法验证，必须在最终说明里明确指出未验证及原因

自动验证 ​

修改 blog/ 下内容后，优先按下面的顺序验证：

  • 先执行 pnpm docs:build，确认 VitePress 可以成功构建，这是最基本的通过条件
  • 如果需要检查页面渲染效果，再执行 pnpm docs:dev
  • 启动 pnpm docs:dev 后，必须使用 agent-browser 以有头模式打开一个可见的测试 Google Chrome 窗口，实际检查页面是否正常显示，是否存在明显的格式问题、资源加载失败或运行时错误提示
  • 测试浏览器必须使用独立临时 profile，不得复用用户默认浏览器配置、登录态、扩展或历史数据
  • 不得仅以 snapshot、DOM 输出、无头截图或命令行结果代替页面预览验证
  • 如果本次修改涉及文案、Markdown 结构、目录、图片、链接或样式，通常应至少补一次页面预览检查
  • 如果因为环境、时间或依赖问题无法完成验证，必须在最终说明中明确写出未验证项和原因
  • 验证完成后，需要彻底关闭打开的浏览器窗口，确保没有遗留的测试环境
