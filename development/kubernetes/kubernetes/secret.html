Kubernetes Secret ​

Secret 用于存储和管理敏感信息，如密码、Token、SSH Key、TLS 证书等。与 ConfigMap 不同，Secret 的数据以 base64 编码存储。

⚠️ Base64 不是加密：默认情况下 Secret 仅以 base64 编码存储在 etcd 中。任何拥有 API 访问权限或 etcd 访问权限的人都可以读取 Secret。生产环境必须启用 静态加密。

Secret 与 ConfigMap 的区别 ​

很多人问：直接把密码放 ConfigMap 不行吗？以下是关键差异：

维度SecretConfigMap
设计目的敏感数据（密码、Token、密钥）非敏感配置（环境变量、配置文件）
存储编码base64 编码存储明文存储
etcd 加密可通过 EncryptionConfiguration 静态加密不适用
kubectl describe隐藏 data 内容，仅显示 key 名称显示全部内容
Volume 挂载写入 tmpfs（内存文件系统），不落磁盘写入节点磁盘
RBAC独立资源 secrets，可单独控制权限独立资源 configmaps
大小限制1 MiB约 1 MiB
自动更新Volume 挂载支持（约 1 分钟延迟）Volume 挂载支持（约 1 分钟延迟）
immutablev1.21+ 支持v1.21+ 支持

核心差异在于安全边界：

shell
# ConfigMap — describe 会暴露内容
kubectl describe configmap db-config
# Data
# ====
# password:
# ----
# 123456          ← 明文可见！

# Secret — describe 不显示内容
kubectl describe secret db-creds
# Data
# ====
# password:  6 bytes   ← 隐藏！

Secret 的 Volume 数据只存在于 tmpfs（内存），永远不会写入节点磁盘。即使 Pod 崩溃，Secret 数据也不会残留在磁盘上。ConfigMap 则会写入节点文件系统。

Secret 类型 ​

类型必需字段用途
Opaque任意 key-value通用密钥，默认类型
kubernetes.io/basic-authusername, passwordHTTP Basic 认证
kubernetes.io/ssh-authssh-privatekeySSH 私钥认证（如 Git 访问）
kubernetes.io/tlstls.crt, tls.keyTLS 证书（Ingress、HTTPS）
kubernetes.io/dockerconfigjson.dockerconfigjson私有镜像仓库认证
kubernetes.io/dockercfg.dockercfg旧版 Docker 仓库认证
kubernetes.io/service-account-tokentoken（自动填充）ServiceAccount 令牌（v1.22+ 推荐 TokenRequest API）
bootstrap.kubernetes.io/tokentoken-id, token-secret节点启动引导令牌（kube-system 命名空间）

自定义类型格式：域名/类型名，如 example.com/cloud-api-credentials。

创建 Secret ​

命令行创建 ​

shell
# Opaque — 从字面量
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='S3cur3P@ss!'

# Opaque — 从文件
kubectl create secret generic db-creds \
  --from-file=./username.txt \
  --from-file=./password.txt

# Opaque — 从 env 文件
kubectl create secret generic app-secrets \
  --from-env-file=.env.production

# SSH 认证
kubectl create secret generic ssh-key \
  --type=kubernetes.io/ssh-auth \
  --from-file=ssh-privatekey=./id_rsa

# TLS 证书
kubectl create secret tls app-tls \
  --cert=./cert.pem \
  --key=./key.pem

# Docker Registry
kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password='S3cur3P@ss!' \
  --docker-email=admin@example.com

YAML 创建 ​

使用 data（base64 编码）：

yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: YWRtaW4=          # echo -n "admin" | base64
  password: UzNjdXIzUEBzcyE=  # echo -n "S3cur3P@ss!" | base64

使用 stringData（明文，创建时自动编码）：

yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  username: admin
  password: "S3cur3P@ss!"
shell
kubectl apply -f secret.yaml

Kustomize 创建 ​

yaml
# kustomization.yaml
secretGenerator:
  - name: app-secrets
    literals:
      - DB_USER=admin
      - DB_PASS=secret

不可变 Secret（v1.21+） ​

yaml
apiVersion: v1
kind: Secret
metadata:
  name: static-config
immutable: true   # 创建后不可修改
type: Opaque
stringData:
  token: "fixed-value"

好处：防止意外修改、减少 kubelet 对 API Server 的 watch 负载。

单个 Secret 大小限制为 1 MiB。

使用 Secret ​

Volume 挂载（推荐） ​

yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-volume-pod
spec:
  containers:
    - name: app
      image: nginx
      volumeMounts:
        - name: secret-volume
          mountPath: /etc/secrets
          readOnly: true
  volumes:
    - name: secret-volume
      secret:
        secretName: db-creds
        defaultMode: 0400         # 文件权限，推荐限制为 0400
        optional: true            # Secret 不存在时 Pod 仍可启动
        items:                    # 选择性挂载指定 key
          - key: password
            path: db-password     # 自定义文件名
            mode: 0400

挂载后每个 key 成为 /etc/secrets/ 下的一个文件：

shell
kubectl exec secret-volume-pod -- ls /etc/secrets
# db-password
# username

kubectl exec secret-volume-pod -- cat /etc/secrets/username
# admin

Volume 挂载的 Secret 会自动更新（约 1 分钟延迟）；环境变量不会自动更新，需重启 Pod。

环境变量（有泄露风险，谨慎使用） ​

单个注入：

yaml
spec:
  containers:
    - name: app
      image: nginx
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-creds
              key: username
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-creds
              key: password

一次性注入全部 key：

yaml
spec:
  containers:
    - name: app
      image: nginx
      envFrom:
        - secretRef:
            name: db-creds

⚠️ 环境变量可能通过 /proc、崩溃转储、调试端点、日志语句泄露。优先使用 Volume 挂载。

Image Pull Secrets ​

拉取私有镜像仓库镜像：

yaml
spec:
  imagePullSecrets:
    - name: regcred
  containers:
    - name: app
      image: private.registry.com/myapp:v1

绑定到 ServiceAccount，自动注入所有使用该 SA 的 Pod：

shell
kubectl patch serviceaccount default \
  -p '{"imagePullSecrets": [{"name": "regcred"}]}'

操作命令 ​

shell
# 查看列表
kubectl get secrets

# 查看 YAML（data 为 base64）
kubectl get secret db-creds -o yaml

# 解码指定 key
kubectl get secret db-creds -o jsonpath='{.data.username}' | base64 -d

# 查看详情（不显示 data 内容）
kubectl describe secret db-creds

# 删除
kubectl delete secret db-creds
kubectl delete -f secret.yaml

安全最佳实践 ​

集群管理员 ​

实践说明
启用静态加密配置 EncryptionConfiguration 加密 etcd 中的 Secret
最小特权 RBAC仅系统组件可 watch/list，用户仅授予命名 Secret 的 get 权限
etcd TLS加密 etcd 实例间通信
审计日志对 Secret 的 create/update/delete 记录 RequestResponse 级别日志
废弃存储擦除下线 etcd 磁盘时物理擦除数据

开发者 ​

实践说明
优先 Volume 挂载避免环境变量泄露到日志和进程元数据
限制文件权限使用 defaultMode: 0400
精确挂载多容器 Pod 中仅挂载到需要的容器
不提交 Git使用 Sealed Secrets / SOPS / External Secrets Operator
不记录日志应用代码不得打印 Secret 值
避免 --from-literal命令行会留在 shell history 中，优先用文件创建

静态加密 ​

shell
# 生成 32 字节密钥
head -c 32 /dev/urandom | base64
yaml
# /etc/kubernetes/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key1
              secret: <base64-encoded-32-byte-key>
      - identity: {}   # 回退，用于读取未加密的旧 Secret

配置 API Server：

shell
kube-apiserver \
  --encryption-provider-config=/etc/kubernetes/encryption-config.yaml

加密全部已有 Secret：

shell
kubectl get secrets --all-namespaces -o json | kubectl replace -f -

RBAC 最小权限 ​

shell
# 仅允许读取指定命名空间下名为 db-creds 的 Secret
kubectl create role secrets-reader \
  --verb=get \
  --resource=secrets \
  --resource-name=db-creds \
  --namespace=production

Secret 轮换策略 ​

  1. 创建版本化新 Secret：db-credentials-v2
  2. 更新 Deployment 引用新 Secret
  3. 滚动重启：kubectl rollout restart deploy/myapp
  4. 验证健康后删除旧 Secret：kubectl delete secret db-credentials-v1

外部 Secret 管理 ​

对于生产级安全需求，可引入外部方案：

方案说明
External Secrets Operator (ESO)从 AWS/GCP/Azure/Vault 同步 Secret 到集群
Secrets Store CSI Driver直接挂载外部 Secret 到 Pod，不创建 K8s Secret 对象
Sealed Secrets (Bitnami)加密 Secret 后可安全存入 Git，仅集群内控制器可解密
HashiCorp Vault全功能 Secret 管理，支持动态 Secret 和自动轮换

常见问题 ​

问题原因/解决
Pod 无法访问 Secret检查命名空间是否正确、RBAC 是否授予 get 权限
更新后 Pod 中未生效Volume 挂载延迟约 1 分钟；环境变量需重启 Pod
subPath 挂载不更新已知限制 — 改用完整目录挂载
大量 Secret 性能问题使用 ResourceQuota 限制数量；考虑不可变 Secret 降低 watch 压力
