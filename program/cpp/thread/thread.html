# C++11 多线程

# 多线程

C++11 中提供了多线程的支持，包括线程的创建、同步、互斥、条件变量等。

概念：

  • 「竞争条件」：多个线程同时访问共享数据，导致数据不一致。最常见的解决竞争条件的方式是「原子操作」，原子操作是不可分割的，不会被其他线程打断，其次是线程同步
  • 「线程同步」：线程同步是指线程间需要按照预定的先后次序顺序进行的行为，比如线程 A 执行完后，线程 B 才能执行，线程 A 和线程 B 之间需要同步，线程同步的目的是为了保护共享数据，防止多个线程同时访问共享数据，导致数据不一致。
  • 「线程同步的方式」：互斥量、读写锁、条件变量、信号量
  • 「临界区」：访问共享资源的代码区域称为临界区 (Critical Sections)

# 线程同步与互斥

自旋锁、互斥量、条件变量、读写锁等都是线程同步的机制，线程同步的目的是为了保护共享数据，防止多个线程同时访问共享数据，导致数据不一致。「互斥量」(mutual exclusive)和「自旋锁」(spinlock)是最底层的两种锁，其他的很多锁都是基于他们的实现。

  • 「自旋锁」加锁失败后，线程会忙等待，直到获取到锁为止，不会阻塞线程，但会占用 CPU 资源。
  • 「互斥量」加锁失败后，线程会陷入休眠，直到获取到锁为止，会阻塞线程，但不会占用 CPU 资源。
  • 「互斥量」和「条件变量」一起使用，互斥量保护共享数据，条件变量用于线程间的通信。

# 自旋锁

while (抢锁(lock) == 没抢到) { /* 自旋 */ }

「自旋锁」尝试加锁失败后，线程会忙等待并占用 CPU资源，直到获取到锁为止。

「自旋锁」的优点是线程不会被阻塞，缺点是线程会一直占用 CPU 资源。

单核 CPU 上，需要抢占式调度器（preemptive scheduler）支持，否则自旋锁会一直占用 CPU 资源，导致其他线程无法运行，造成「线程饥饿」

「自旋锁」是通过 CPU 提供的 CAS (Compare And Swap) 指令实现的，CAS 指令是原子操作，所以自旋锁是线程安全的。[1]

「自旋锁」在「用户态」完成加锁和解锁操作，不会主动产生线程的上下文切换，效率较高

# 互斥量

while(抢锁(lock) == 没抢到) {
  线程进入休眠，在这把锁的状态发生改变时唤醒(lock);
}

「互斥量」尝试加锁失败后，会释放 CPU 资源陷入休眠，直到被唤醒，这个过程中：

  • 线程加锁失败，会从用户态陷入内核态，内核会把线程从「运行」状态切换到「休眠」状态，然后把线程放到等待队列中
  • 当互斥量的锁被释放时，内核会从等待队列中唤醒一个线程，把线程从「休眠」状态切换到「运行」状态，这个过程需要从用户态切换到内核态，然后再从内核态切换到用户态，这个过程需要消耗一定的开销。

线程的上下文切换：多个线程同属于一个进程，其虚拟内存是共享的，因此在切换时，不需要切换虚拟内存，只需要切换线程的上下文，比如私有数据、寄存器、栈等不共享数据。

上下文切换的的耗时（可能）大概在几十纳秒到几微秒之间，如果锁住的代码执行时间很短，那么上下文切换的时间可能比执行代码时间还长。因此，如果锁住的代码执行时间很短，就应该使用自旋锁，而不是互斥锁。

在 C++11 中，互斥量 (#include <mutex>)的实现有两种，分别是「互斥量」std::mutex 和「递归互斥量」std::recursive_mutex，两者的区别如下：

  • std::mutex 是非递归的，std::recursive_mutex 是递归的
  • std::mutex 不能被多次加锁，std::recursive_mutex 可以被多次加锁
  • std::mutex 不能被多次解锁，std::recursive_mutex 可以被多次解锁

C++11 中对互斥量的管理模板类有三种，分别是 std::lock_guard、std::unique_lock 和 std::scoped_lock。std::lock_guard 是 std::unique_lock 的简化版，std::scoped_lock 是 std::unique_lock 的升级版

lock_guard 在构造函数中对互斥量加锁，在析构函数中对互斥量解锁，没有提供其他的函数

unique_lock 可以手动加锁 (lock()) 和解锁 (unlock()) ，scoped_lock 可以同时管理多个互斥量。

std::lock_guard<std::mutex> lock1(m1);
lock1.unlock(); // error
std::unique_lock<std::mutex> lock2(m2);
lock2.lock();   // ok

# 条件变量

在互斥量中提到，休眠的线程会被内核唤醒，而条件变量就是用于线程间的通信的，当条件变量满足时，线程会被唤醒。因此条件变量通常是与互斥量一起使用的，互斥量保护共享数据，条件变量用于线程间的通信。

条件变量本质上也是一个多线程间共享的全局变量，它的功能是阻塞线程，被阻塞的线程直到接收到“条件成立”的信号后才能继续执行。

  • 条件变量并不是锁（但它几乎总是和互斥量一起使用的），而是线程间的一种通讯机制
  • 条件变量本身也不包含条件，它被称为条件变量是因为它经常和条件语句（if/while）一起使用

C++11 中，条件变量类是 std::condition_variable，其构造函数接受一个互斥量，用于保护条件变量，条件变量的 wait 函数接受一个互斥量和一个可调用对象，当条件变量满足时，wait 函数返回，否则 wait 函数会解锁互斥量，然后阻塞当前线程，直到条件变量满足时，wait 函数会重新加锁互斥量，然后返回。

#include <condition_variable>

成员函数主要包括 wait、notify_one 和 notify_all，其中 wait 函数有两个版本，一个版本接受一个互斥量，一个版本接受一个互斥量和一个可调用对象。

void wait(unique_lock<mutex>& __lk) _NOEXCEPT;
template <class _Predicate>
    _LIBCPP_METHOD_TEMPLATE_IMPLICIT_INSTANTIATION_VIS
    void wait(unique_lock<mutex>& __lk, _Predicate __pred);

# 读写锁

读写锁是一种特殊的锁，允许多个线程同时「读」共享数据，但是只允许一个线程「写」共享数据。

  • 当「写锁」没有被线程持有时，多个线程能够并发地持有「读锁」，并且能够同时访问共享数据。
  • 当「写锁」被线程持有时，其他线程不能持有「读锁」，也不能持有「写锁」

读优先锁希望的是读锁能够被更多的线程获取，可以提高读线程的并发性。线程A获取了读锁，线程B想获取写锁，此时会被阻塞，线程c可以继续获取读锁，直到A和c释放锁，线程B才可以获取写锁。如果有很多线程获取读锁，且加锁的代码执行时间很长，就到导致线程B永远获取不到写锁。写优先锁希望的是写锁能够被优先获取。线程A获取了读锁，线程B想获取写锁，此时会被阻塞，后面获取读锁都会失败，线程A释放锁，线程B可以获取写锁，其他获取读锁的线程阻塞。如果有很多写线程获取写锁，且加锁的代码执行时间很长，就到导致读线程永远获取不到读锁。上面两种锁都会造成【饥饿】现象，为解决这种问题，可以增加一个队列，把获取锁的线程（不管是写线程还是读线程）按照先进先出的方式排队，每次从队列中取出一个线程获取锁，这种获取锁的方式是公平的。

# 信号量

信号量分为有名信号量和无名信号量，「无名信号量」用于线程同步，有名信号量一般用于进程之间管理。

信号量本质上是一个非负的整数计数器，用于控制公共资源的访问，也被称为PV原子操作：

  • P操作：即信号量sem减一，若sem小于等于0则P操作被阻塞，直到sem变量大于0为止
  • V操作：即信号量sem加一

信号量允许多个线程同时进入临界区，而互斥量只允许一个线程进入临界区。

# 乐观锁和悲观锁

互斥锁、自旋锁、读写锁，都是属于悲观锁，因为它们都是假设线程会发生冲突，所以在加锁时，会阻塞线程，等待其他线程释放锁。

乐观锁是假设线程不会发生冲突，所以在加锁时，不会阻塞线程，而是直接访问共享数据，如果发生冲突，则重试。

乐观锁是先修改共享资源，再用历史数据和当前数据比对验证这段时间共享数据有没有被修改，如果没有被修改，那么更新数据，如果有其他线程更新了共享资源，需要重新获取数据，再更新，验证，循环往复的重试，直到更新成功。所以当数据更新操作比较频繁，数据冲突的概率就会比较大，重试的次数就会多，浪费CPU资源。

乐观锁其实全程没有加锁，也叫无锁编程，所以针对读多写少的场景，并发性能较高，典型的实现MVCC，mysql中会使用MVCC构建一致性读来保证可重复读。悲观锁是在访问共享资源之前统统加锁。当并发冲突概率较高时，乐观锁不在适用，悲观锁就排上用场。互斥锁、自旋锁都是悲观锁的实现。

# 线程死锁

在许多应用中进程需要以独占的方式访问资源，当操作系统允许多个进程并发执行时可能会出现进程永远被阻塞现象，如两个进程分别等待对方所占的资源，于是两者都不能执行而处于永远等待状态，此现象称为死锁。

死锁通常被定义为：如果一个进程集合中的每个进程都在等待只能由此集合中的其他进程才能引发的事件，而无限期陷入僵持的局面称为死锁。

死锁的四个条件同时：

  1. 「互斥条件」临界资源是独占资源，进程应互斥且排他的使用这些资源
  2. 「请求与保持条件」进程已经保持了至少一个资源，但又提出了新的资源请求时，该资源已被其他进程占有，此时请求进程被阻塞，但对自己已获得的资源保持不放。
  3. 「不可剥夺条件」又称不可抢占，已获资源只能由进程自愿释放，不允许被其他进程剥夺
  4. 「循环等待条件」又称环路条件，存在循环等待链，其中，每个进程都在等待链中等待下一个进程所持有的资源，造成这组进程处于永远等待状态

死锁产生的原因有：

  • 进程顺序不当
  • PV操作使用不妥当
  • 同类资源分配不均
  • 对某些资源的使用未加限制
  • ...

产生死锁的原因不仅与系统拥有的资源数量有关，而且与资源分配策略、进程对资源的使用要求以及进程的推进顺序有关。

死锁解决的方法：

  • 「死锁防止」破坏死锁产生的四个条件之一
  • 「死锁避免」破坏死锁产生的四个条件之一
  • 「死锁检测和恢复」破坏死锁产生的四个条件之一

# 死锁示例

std::mutex mutex1;
std::mutex mutex2;
void func1() {
    std::cout << "func1 wait for mutex1" << std::endl;
    std::unique_lock<std::mutex> lock1(mutex1);
    std::cout << "func2 get mutex1" << std::endl;

    std::this_thread::sleep_for(std::chrono::seconds(2));

    std::cout << "func1 wait for mutex2" << std::endl;
    std::unique_lock<std::mutex> lock2(mutex2);
    std::cout << "func2 get mutex1" << std::endl;
}
void func2() {
    std::cout << "func2 wait for mutex2" << std::endl;
    std::unique_lock<std::mutex> lock2(mutex2);
    std::cout << "func2 get mutex2" << std::endl;

    std::this_thread::sleep_for(std::chrono::seconds(2));

    std::cout << "func2 wait for mutex1" << std::endl;
    std::unique_lock<std::mutex> lock1(mutex1);
    std::cout << "func2 get mutex1" << std::endl;
}
int main(int argc, char const* argv[]) {
    std::thread t1(func1);    std::thread t2(func2);
    t1.join();                t2.join();
    return 0;
}

# 死锁防止

破坏死锁产生的四个条件之一

  1. 「破坏互斥条件」允许多个进程同时访问资源，如打印机可同时服务于多个进程

    只读数据文件、磁盘等软硬件资源均可采用这种办法管理；
    但是许多资源是独占性资源，如可写文件、键盘等只能互斥的占有；
    所以这种做法在许多场合是不适用的。

  2. 「破坏占有和等待条件」采用静态分配的方式，静态分配的方式是指进程必须在执行之前就申请需要的全部资源，且直至所要的资源全部得到满足后才开始执行。

    实现简单，但是严重的减低了资源利用率。
    因为在每个进程占有的资源中，有些资源在运行后期使用，有些资源在例外情况下才被使用，可能会造成进程占有一些几乎用不到的资源，而使其他想使用这些资源的进程等待。

  3. 「破坏不剥夺条件」当一个进程占有一些资源后，又提出了新的资源请求而得不到满足时，它必须释放已占有的全部资源，待以后需要时再重新申请。

    剥夺调度能够防止死锁，但是只适用于内存和处理器资源。
    方法一：占有资源的进程若要申请新资源，必须主动释放已占有资源，若需要此资源，应该向系统重新申请。
    方法二：资源分配管理程序为进程分配新资源时，若有则分配；否则将剥夺此进程已占有的全部资源，并让进程进入等待资源状态，资源充足后再唤醒它重新申请所有所需资源。

  4. 「破坏循环等待条件」给系统的所有资源编号，规定进程请求所需资源的顺序必须按照资源的编号依次进行。

    采用层次分配策略，将系统中所有的资源排列到不同层次中

# 死锁避免

所谓银行家算法，是指在分配资源之前先看清楚，资源分配后是否会导致系统死锁。如果会死锁，则不分配，否则就分配。[9]

按照银行家算法的思想，当进程请求资源时，系统将按如下原则分配系统资源：

  1. 当一个进程对资源的最大需求量不超过系统中的资源数时可以接纳该进程。
  2. 进程可以分期请求资源，当请求的总数不能超过最大需求量。
  3. 当系统现有的资源不能满足进程尚需资源数时，对进程的请求可以推迟分配，但总能使进程在有限的时间里得到资源。
  4. 当系统现有的资源能满足进程尚需资源数时，必须测试系统现存的资源能否满足该进程尚需的最大资源数，若能满足则按当前的申请量分配资源，否则也要推迟分配。

# 线程池

线程池主要解决两个问题：

  1. 线程创建与销毁的开销以及线程竞争造成的性能瓶颈。通过预先创建一组线程并复用它们，线程池有效地「降低了线程创建和销毁的时间和资源消耗」。
  2. 过多的线程可能导致线程竞争，影响系统性能。线程池通过管理线程并发数量，线程池有助于「减少线程之间的竞争」，增加资源利用率，并提高程序运行的性能。例如，当CPU密集型任务和I/O密集型任务共存时，可以通过调整线程池资源，实现更高效的负载平衡。

线程池：管理一个任务队列，一个线程队列，然后每次取一个任务分配给一个线程去做，循环往复

# 原子操作

原子操作就是: 不可中断的一个或者一系列操作, 也就是不会被线程调度机制打断的操作, 运行期间不会有任何的上下文切换(context switch).

意义:

  • 如果确定某个操作是原子的, 就不用为了去保护这个操作而加上会耗费昂贵性能开销的锁. - (巧妙的利用原子操作和实现无锁编程)
  • 借助原子操作可以实现互斥锁(mutex)

# 简单线程同步

线程的创建使用 std::thread 类，其构造函数接受一个可调用对象，创建一个线程并执行该可调用对象。

多线程的正确运行可能需要一些同步机制，比如互斥量、条件变量等，这些同步机制需要在主线程中创建，然后传递给子线程。

以「生产者-消费者」为例，主线程创建一个互斥量和一个条件变量，然后创建两个子线程，一个作为生产者，一个作为消费者，两个子线程都需要使用互斥量和条件变量，所以需要将这两个同步机制传递给子线程。

其中

  • 「互斥量」是用于保护共享数据的，比如生产者和消费者共享的队列，生产者在向队列中插入数据时需要加锁，消费者在从队列中取出数据时也需要加锁；
  • 「条件变量」是用于线程间的通信的，比如生产者在向队列中插入数据时，如果队列已满，则生产者需要等待，直到队列不满时，消费者从队列中取出数据时，如果队列为空，则消费者需要等待，直到队列不空时。

与线程相关的头文件如下：

#include <thread>
#include <mutex>
#include <condition_variable>

生产者线程的函数如下：

void producer(
    std::queue<int>& q, // 队列
    std::mutex& m,      // 互斥量
    std::condition_variable& cv // 条件变量
    ) {
    for (int i = 0; i < 10; ++i) {
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
        std::unique_lock<std::mutex> lock(m);
        q.push(i);
        std::cout << "push " << i << std::endl;
        cv.notify_one();
    }
}

  • std::this_thread::sleep_for 用于让当前线程休眠一段时间

  • std::unique_lock 用于对互斥量加锁，std::unique_lock 的构造函数接受一个互斥量，构造函数中对互斥量加锁，析构函数中对互斥量解锁，所以可以使用 std::unique_lock 对互斥量加锁，当 std::unique_lock 对象销毁时，会自动解锁互斥量

  • 当前线程对互斥量加锁后，可以对共享数据进行操作，比如向队列中插入数据，然后通过 std::condition_variable 中的 notify_one 通知消费者线程，队列中已经有数据了，可以从队列中取出数据了

    • notify_one 函数用于通知一个等待的线程，notify_all 函数用于通知所有等待的线程。

消费者线程的函数如下：

void consumer(std::queue<int>& q, std::mutex& m, std::condition_variable& cv) {
    for (int i = 0; i < 10; ++i) {
        std::unique_lock<std::mutex> lock(m);
        cv.wait(lock, [&q] { return !q.empty(); });
        int val = q.front();
        q.pop();
        std::cout << "pop " << val << std::endl;
    }
}

  • 在消费者线程中，同样 std::unique_lock<std::mutex> lock(m); 对互斥量加锁，

  • std::condition_variable 是条件变量类，其构造函数接受一个互斥量，用于保护条件变量，条件变量的 wait 函数接受一个互斥量和一个可调用对象，当条件变量满足时，wait 函数返回，否则 wait 函数会解锁互斥量，然后阻塞当前线程，直到条件变量满足时，wait 函数会重新加锁互斥量，然后返回。

主线程中创建互斥量和条件变量，然后创建两个子线程，最后等待两个子线程结束。

int main() {
    std::queue<int> q;
    std::mutex m;
    std::condition_variable cv;
    std::thread t1(producer, std::ref(q), std::ref(m), std::ref(cv));
    std::thread t2(consumer, std::ref(q), std::ref(m), std::ref(cv));
    t1.join();
    t2.join();
    return 0;
}
  • join 和 detach 所有权移交：
    • join 用于阻塞主线程，主线程等待线程结束，然后回收线程资源
    • detach 用于将线程资源移交给系统，驻留后台，系统会负责回收线程资源
    • 一个线程只能调用一次 join 或 detach，否则会抛出异常
    • 如果线程没有调用 join 或 detach，则会在析构时调用 std::terminate 终止程序
  • join 和 detach 的区别：
    • join 会阻塞当前线程，直到被调用的线程结束，detach 不会阻塞当前线程
    • join 会等待被调用的线程结束，detach 不会等待被调用的线程结束
    • join 会回收被调用的线程资源，detach 不会回收被调用的线程资源

销毁线程前需要做的事：线程可能会访问共享数据，所以需要等待线程结束，然后回收线程资源
