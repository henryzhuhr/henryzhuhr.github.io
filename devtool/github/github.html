GitHub Pages ​

官方文档 GitHub Pages 使用入门

官方对 GitHub Pages 的定义是：GitHub Pages 是一项静态站点托管服务，它直接从 GitHub 上的仓库获取 HTML、CSS 和 JavaScript 文件，通过构建过程运行文件，然后发布网站。

因此，GitHub Pages 适合用于构建个人博客、项目文档、展示页等静态网站。

GitHub Pages 会使用一个 <username>.github.io 域名作为网页的地址，并且支持如下几种站点类型：

  • 用户站点：<username>.github.io

用户站点需要创建一个名为 <username>.github.io 的仓库，该仓库会自动部署到 http(s)://<username>.github.io 网站上，这个站点是唯一的，并且和用户名绑定的，一般来说会部署个人站点或个人简介

  • 项目站点：<username>.github.io/<repository>

GitHub 也为每一个仓库提供了一个独立的站点，可以通过 <username>.github.io/<repository> 访问。当为仓库开启 GitHub Pages 服务后，该仓库的 Pages 分支会自动部署到 http(s)://<username>.github.io/<repository> 网站上，这个站点是独立的，和仓库绑定的，一般用于项目文档、展示页等

开启 GitHub Pages 服务 ​

在仓库的 Settings 页面中找到 Pages 选项，可以启用相关功能，并选择要部署的分支和目录即可，例如选择 gh-pages 分支，/ 目录，然后点击 Save 按钮即可，随后当检测到被选择分支更新的时候，GitHub Pages 会自动部署新的网站，这个过程可以在 Actions 页面中查看

到此，会有一个疑问，gh-pages 哪来的？

实际上，GitHub Pages 并不是单独工作的，一般来说会使用「静态站点生成器」（例如Vitepress 等）结合 GitHub Actions 功能完成持续集成(Continuous Integration, CI)，使用 GitHub Pages 完成持续部署(Continuous Deployment, CD)的工作，这样就可以实现自动化构建和部署静态网站，一般流程如下：

  1. 静态站点生成器生成站点模板，在 docs 目录中写文章
  2. 使用 Github Actions 生成/编译出静态站点(html/js/css)
  3. 使用 GitHub Pages 自动部署网页

例如可以参考 Vitepress 的文档快速生成一个静态网站模板

基于 Vitepress 的 GitHub Pages 部署 ​

Vitepress 只是多个「静态站点生成器」的一个，还有 gitbook 等，具体如何使用请详细阅读对应的文档。但是大致流程是一样的

  1. 创建一个仓库，并 git clone 到本地，进入仓库目录

  2. 需要安装 node.js，并且确保 npm 是可用的

  3. 创建一个 Vitepress 项目，这些步骤具体请参考官方指南 Vitepress 快速开始，后面只说一些要点

    • 推荐使用 pnpm 而不是 npm 进行后续步骤，但是需要安装 pnpm：npm install -g pnpm
    • 在项目目录内，安装 Vitepress pnpm add -D vitepress
    • 初始化 vitepress pnpm vitepress init ，选择一些初始化配置
    • 生成项目模板后，就可以在 docs 中编写自己的文章，编写过程中可以 pnpm docs:dev 随时预览
    • 编写完成后 pnpm docs:build 查看编译是否有问题

以上部分请详细阅读官方文档，完整站点的相关配置，例如生成的站点需要根据仓库名称 设定 public 根目录

  1. 参考 Vitepress 部署 - GitHub Pages，创建 .github/workflows/deploy.yml 文件

.github/workflows 是 Github Actions 工作流文件，在上传代码满足特定条件，例如 on.push.branch: [main] 表示推送代码到 main 分支后自动执行构建任务

根据 vitepress 提供的 .github/workflows/deploy.yml ，包含了两个任务(jobs)：build 和 deploy

yaml
jobs:
  build:
    # ...
  deploy:
    # ...
  • build 任务是编译站点，实际上和在本地执行 pnpm docs:build 是一样的，
yml
  # 构建工作
  build:
    runs-on: ubuntu-latest # 表示构建工作在 ubuntu 系统下完成
    steps:  # 构建工作的步骤，
      - name: Checkout  
        uses: actions/checkout@v4
      - name: Setup Pages 
        uses: actions/configure-pages@v4
      - name: Install dependencies  # 安装依赖`pnpm install`
        run: pnpm install
      - name: Build with VitePress  # 编译站点`pnpm docs:build`
        run: pnpm docs:build
      - name: Upload artifact       # 编译的目录
        uses: actions/upload-pages-artifact@v3
        with:
          path: docs/.vitepress/dist
  • deploy 任务是部署站点到 gh-pages 分支，具体如何部署，可以参考使用的 actions/deploy-pages@v4
yml
  # 部署工作
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    needs: build
    runs-on: ubuntu-latest
    name: Deploy
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
  1. 当推送代码后指定分支，会自动触发 actions，等待完成后就可以到 http(s)://<username>.github.io 或 http(s)://<username>.github.io/<repository> 查看部署的网页了

如果能看到站点文字内容，看不到页面的样式，有可能没有正确设置 base 参数，参考 设定 public 根目录。如果是用户站点的仓库 user.github.io，可以不需要设置；但是如果是 user.github.io/repo/ 项目仓库，则需要设置 base: '/repo/'

Github Action ​

自动同步fork的上游分支 ​

yml
# .github/workflows/sync-fork.yml
name: Sync Fork

on:
  schedule:
    - cron: '0 0 * * *' # every day
  workflow_dispatch: # on button click

jobs:
  sync:

    runs-on: ubuntu-latest

    steps:
      - uses: tgymnich/fork-sync@v1.9.0
        # https://github.com/tgymnich/fork-sync
        with:
          owner: meta-llama # 上有分支仓库所述者
          base: main
          head: main
